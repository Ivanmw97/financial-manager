## 1. Project Overview

**Financial Manager** is a personal finance SPA (Single Page Application) that allows users to:

- Record and categorise income/expense transactions
- Define monthly spending budgets per category
- Visualise financial stats (totals, category breakdown, net savings)
- Operate in **Guest Mode** (localStorage only, no account required)
- Operate as an **Authenticated User** (data persisted in Supabase)

Deployed to **GitHub Pages**: `https://ivanmw97.github.io/financial-manager`

## 4. Data Model

### `Transaction`
```ts
interface Transaction {
  id: string;            // UUID (generated by Supabase or crypto.randomUUID in guest mode)
  type: 'income' | 'expense';
  description: string;
  category: string;      // Free text; loosely matched against Budget.category
  amount: number;        // Always positive
  date: string;          // ISO date string: "YYYY-MM-DD" or "YYYY-MM-DDTHH:mm:ssZ"
}
```

### `Budget`
```ts
interface Budget {
  id?: string;
  user_id?: string;
  category: string;       // Must match Transaction.category for spending calc
  budget_limit: number;   // Monthly spending cap for this category
  month?: string;         // "YYYY-MM" — the month this budget applies to
  created_at?: string;
}
```

> **Important:** The Supabase DB column is named `budget_limit` (not `limit`). `database.types.ts` reflects this. The `month` field in `budgets` defaults to the current month (`budgetStore.currentMonth`).

### Supabase Tables

| Table | Key Columns |
|---|---|
| `transactions` | `id`, `user_id`, `type`, `amount`, `date`, `category`, `description` |
| `budgets` | `id`, `user_id`, `category`, `budget_limit`, `month` |
| `profiles` | `id` (= auth.users.id), `email`, `first_name`, `last_name`, `currency`, `theme` |
| `categories` | `id`, `user_id`, `name`, `type`, `color`, `icon` *(not yet used in UI)* |

Row Level Security (RLS) is enforced on all tables via `user_id = auth.uid()`.

## 5. Business Logic

### Authentication Flow

```
App mount (App.vue onMounted)
  └─ userStore.initialize()
       ├─ Check localStorage for 'guestMode' === 'true' → set isGuestMode = true
       └─ supabase.auth.getUser() → set user if session exists
           └─ if user OR isGuestMode → load transactions + budgets in parallel
```

Navigation guards in `router/index.ts` wait for `isLoading` to resolve before evaluating auth state. Routes are tagged with `meta.requiresAuth` or `meta.requiresGuest`.

### Guest Mode

- Users can bypass authentication entirely.
- All data is stored in `localStorage` under keys `guest_transactions` and `guest_budgets`.
- On sign-in, guest data is **not** migrated to the authenticated account — it is cleared.
- Guest mode state is persisted via `localStorage.setItem('guestMode', 'true')`.

### Dual Storage Strategy (Stores)

Every store action (`addTransaction`, `updateBudget`, `removeTransaction`, etc.) branches on `userStore.isGuestMode`:

```
isGuestMode === true  → read/write localStorage
isGuestMode === false → read/write Supabase (requires userStore.user.id)
```

### Budget Spending Calculation

Spending per budget category is **not stored** — it is always computed at runtime:

```ts
spent = transactions
  .filter(t => t.type === 'expense' && t.category === budget.category && t.date.startsWith(currentMonth))
  .reduce((sum, t) => sum + t.amount, 0)
```

`budget_limit` and `spent` are combined into `BudgetWithSpent` objects in Dashboard and Budgets views via computed properties.

### Monthly Reset Behaviour

Budgets are **not deleted** at month end. The `currentMonth` state in `budgetStore` (format: `YYYY-MM`) acts as the filter. When `checkMonthlyReset()` is called and detects a new month, it reloads budgets. Spending naturally resets because the transaction filter only counts the current month's expenses.

### Password Reset Flow

Supabase sends an email with a deep-link containing `type=recovery` in the URL hash. The router guard allows access to `/reset-password` when this hash is present. `ResetPassword.vue` handles the actual `supabase.auth.updateUser({ password })` call.

## 6. State Management (Pinia Stores)

### `useUserStore` (`store/user.ts`)
| State | Type | Description |
|---|---|---|
| `user` | `User \| null` | Supabase auth user object |
| `isLoading` | `boolean` | True during async auth operations |
| `isGuestMode` | `boolean` | Whether running in offline guest mode |

Key actions: `initialize()`, `signIn()`, `signUp()`, `signOut()`, `setGuestMode()`.

### `useTransactionStore` (`store/transactions.ts`)
| State | Type | Description |
|---|---|---|
| `transactions` | `Transaction[]` | Full list (current user or guest) |
| `isLoading` | `boolean` | True during Supabase fetches |

Getters: `getTransactionsByMonth(month)`, `availableMonths`, `currentMonthTransactions`.

### `useBudgetStore` (`store/budgets.ts`)
| State | Type | Description |
|---|---|---|
| `budgets` | `Budget[]` | All budgets for current user |
| `currentMonth` | `string` | `YYYY-MM` of the current month |
| `isLoading` | `boolean` | True during Supabase fetches |

### `useModalStore` (`store/modal.ts`)
Simple boolean store (`isOpen`) used for add-transaction modal visibility. Currently lightly used.

## 7. Routing

| Path | Component | Guard |
|---|---|---|
| `/` | — | Redirects to `/auth` |
| `/auth` | `Auth.vue` | `requiresGuest` (redirects to `/dashboard` if already auth'd) |
| `/auth/callback` | `AuthCallback.vue` | Public |
| `/reset-password` | `Auth.vue` + `ResetPassword.vue` (child) | Public (special-cased for recovery hash) |
| `/dashboard` | `Dashboard.vue` | `requiresAuth` |
| `/transactions` | `Transactions.vue` | `requiresAuth` |
| `/budgets` | `Budgets.vue` | `requiresAuth` |
| `/stats` | `Stats.vue` | `requiresAuth` |
| `/:pathMatch(.*)` | — | Redirects to `/dashboard` |

Router uses `createWebHistory` with base `/financial-manager/` (GitHub Pages subdirectory). A `404.html` in `/public` handles SPA deep-link redirects on GitHub Pages.

## 8. Environment Variables

| Variable | Required | Description |
|---|---|---|
| `VITE_SUPABASE_URL` | Yes | Supabase project URL (`https://<id>.supabase.co`) |
| `VITE_SUPABASE_ANON_KEY` | Yes | Supabase anonymous/public key |

Copy `.env.example` to `.env` to get started. Variables are exposed to the client via Vite's `import.meta.env`.

> **Security note:** The anon key is intentionally public-facing. Supabase RLS policies enforce data isolation per `user_id`. Never put the `service_role` key in a frontend env file.

## 11. Known Limitations & Future Work

| Area | Issue / Opportunity |
|---|---|
| `Stats.vue` | "Daily Spending Trend" chart is a placeholder (random bar heights, not real data) |
| `categories` table | Defined in DB schema but not wired to the UI — categories are free text strings |
| `profiles` table | Created on sign-up, read in `UserProfile.vue`, but `currency` and `theme` preferences are not applied to the UI |
| Guest → Auth migration | Logging in from guest mode silently drops guest data instead of offering to import it |
| Error handling | No global error boundary; Supabase failures surface only via `console.error` |
| Chart.js | Imported as a dependency but only used in `Stats.vue` placeholder — real charts not implemented yet |
| `modal.ts` store | Exists but the add-transaction modal in `Transactions.vue` uses a local `ref` instead |
